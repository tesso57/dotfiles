## Context

- `cmux-delegate` は cmux の IPC プリミティブを組み合わせた委任スクリプト
- 別ペインで AI エージェントや任意コマンドを起動し、完了を待って結果を収集する
- エージェント種別: `claude`（対話型）、`codex`（対話型）、`gemini`（ワンショット）、`cmd`（任意コマンド）

## 環境チェック

まず cmux 環境で実行されているか確認する:
```bash
cmux ping
```
失敗した場合は「cmux 環境外で実行されています」と伝えて終了する。

## エージェントプロファイル

全フェーズで参照するリファレンス。タイムアウト・ポーリング・クリーンアップ等の設定はこの表に従う。

| 特性 | claude | codex | gemini | cmd |
|---|---|---|---|---|
| デフォルトタイムアウト | 900s | 3600s（終了通知まで粘る） | 180s | 180s |
| 推奨ポーリング間隔 | 60s | 30s | N/A | N/A |
| 完了検出 | done シグナル | done シグナル + `›` idle 検出 | done シグナル | done シグナル |
| 結果取得 | read-screen 200行 | read-screen 200行 | /tmp ファイル | /tmp ファイル |
| フリーズ判定 | 60s 画面変化なし | 60s 画面変化なし + Working なし | ファイルサイズ不変 | ファイルサイズ不変 |
| クリーンアップ猶予 | 3s | 5s | 0s | 0s |

## モード判定

`$ARGUMENTS` を分析してモードを決定する:

### 1. 明示指定

| 先頭トークン | モード | 例 |
|---|---|---|
| `commander` | Commander モード | `/delegate commander "認証機能を実装して"` |
| `single` | Single モード（強制） | `/delegate single codex "リファクタ"` |

### 2. エージェント指定（Single モード）

先頭トークンが `claude` / `codex` / `gemini` / `cmd` の場合 → Single モード

### 3. タスク性質推論

先頭トークンが上記のいずれにも一致しない場合、プロンプト全体を分析して最適なモードを推論する:

| キーワード・パターン | 推論結果 |
|---|---|
| 調査/探索/検索/一覧/構成/どうなっている | Commander + Divide |
| 比較/設計/どちらが/選定/アーキテクチャ | Commander + Brainstorm |
| レビュー/チェック/確認/品質/問題ないか | Commander + Review |
| 複数の独立作業を含む（「AとBをやって」等） | Commander + Divide |
| 上記に該当しない単一タスク | Single |

推論時は判定理由を 1 行報告してから実行する:
> モード判定: 「調査」キーワードを検出 → Commander + Divide を選択

## Single モード

### 引数パース

`$ARGUMENTS` を解析する（`single` プレフィックスがあれば除去してから）:
- 形式: `[agent_type] <prompt>`
- 先頭トークンが `claude` / `codex` / `gemini` / `cmd` のいずれかならそれをエージェント種別とし、残りをプロンプトとする
- いずれにも一致しない場合、全体をプロンプトとして `claude` をデフォルトにする

例:
- `/delegate claude "構成を説明して"` → agent=claude
- `/delegate codex "リファクタして"` → agent=codex
- `/delegate cmd "go test ./..."` → agent=cmd
- `/delegate "構成を説明して"` → agent=claude (デフォルト)

### タスク性質評価

引数パース後、実行フローの前にタスクの性質を評価する。

**重要**: この評価は**モード判定ステップ 2（エージェント指定）で Single に確定した場合のみ**適用される。ステップ 3 の推論で Commander に振り分けられた場合は、Single の Parallel 判定には進まない。

**並列探索フロー（Parallel）**:
- 条件: 調査/探索/比較系のタスク **かつ** agent_type が `claude` または `codex`
- Agent A: 直接的アプローチでタスクを実行
- Agent B: 関連コンテキスト探索（例: 「対象ファイルの git log、import 元、呼び出し元を調査し、影響範囲を報告せよ」）
- 両方の結果を統合して報告

**逐次フロー（Serial）**:
- 条件: 上記以外（実装系タスク、cmd、gemini、明確な単一作業）
- 従来の 1 エージェントフロー

### 実行フロー — Serial

#### 1. エージェント起動

```bash
cmux-delegate start <agent_type> "<prompt>"
```

出力をパースして TASK_ID と SURFACE_REF を取得する:
```
TASK_ID=delegate-1709312345-12345
SURFACE_REF=surface:2
```

ユーザーに起動完了を報告する。

#### 2. 完了待機

エージェントプロファイル表のタイムアウト・ポーリング間隔を使用する。**`gemini` / `cmd` はポーリング不要のため `wait` を使用する。**

**claude / codex（対話型）**:
```bash
cmux-delegate wait-poll <task_id> <timeout> <interval> <surface_ref> <agent_type>
```
- `PROGRESS` 出力をユーザーに進捗テーブルとして提示する（バックグラウンド実行を推奨）
- 完了時: → 結果取得へ
- タイムアウト時のフォールバック:
  1. `cmux-delegate probe <task_id> <surface_ref> <agent_type>` で状態確認
  2. `active` → 追加 60s 待機（`wait-poll` を 1 回のみ再実行）
  3. `frozen` / `stalled` → `read` で途中結果を取得 → `cleanup`
  4. `completed` / `idle` → `read` → `cleanup`
  5. `unknown` → `read` 試行 → `cleanup`
- frozen 検出（return 2）: 即座に `read` で途中結果 → `cleanup`

**gemini / cmd（ワンショット）**:
```bash
cmux-delegate wait <task_id> <timeout>
```
- ブロック待ちのみ。ポーリング不要。

#### 3. 結果取得

```bash
cmux-delegate read <task_id> [surface_ref] [lines]
```

- `cmd` / `gemini`: `/tmp/<task_id>.out` から自動読み取り（surface_ref 不要）
- `claude` / `codex`: surface_ref を使ってスクリーンバッファから読み取り
- 結果末尾が不完全（文の途中で切れている等）な場合:
  - `claude` / `codex`: 追加 5s 待ち + 再 read

#### 4. 結果分析と報告

取得した結果を分析し、ユーザーにわかりやすく報告する:
- 要約を提示
- エラーがあれば指摘
- 追加アクションが必要であれば提案

#### 5. クリーンアップ

```bash
cmux-delegate cleanup <task_id> [surface_ref] [grace_period]
```

エージェントプロファイル表の猶予秒数を `grace_period` に渡す。
surface_ref を渡すとペインも閉じる。省略時は一時ファイルのみ削除。

### 実行フロー — Parallel（並列探索）

#### 1. 2 エージェント起動

```bash
# Agent A: 直接アプローチ
cmux-delegate start <agent_type> "<直接的な指示プロンプト>"
# → TASK_ID_A, SURFACE_REF_A

# Agent B: コンテキスト探索
cmux-delegate start <agent_type> "<関連コンテキスト探索プロンプト>"
# → TASK_ID_B, SURFACE_REF_B
```

起動状況をユーザーにテーブル形式で報告する。

#### 2. 並列待機 + 結果収集

各タスクに対して `wait-poll` → `read` を順次実行する（Serial と同じフォールバック適用）。

#### 3. 統合報告

- Agent A の結果を主軸とし、Agent B の発見で補完する
- 矛盾がある場合は両方を提示して違いを説明する

#### 4. クリーンアップ

全タスクに対して `cleanup`（エージェントプロファイルの猶予を適用）。

## Commander モード

`/delegate commander "<タスク記述>"` で起動する。
あなたは **コマンダー（指揮官）** として、タスクを分析し最適な戦略を自律的に選択・実行する。

### ステップ 1: タスク分析

タスクの性質を分析し、以下を整理する:

- **種類**: 実装 / 調査 / 設計 / レビュー / アイデア出し / テスト
- **規模**: 小（1 エージェント） / 中（2〜3 並列） / 大（4 並列）
- **成果物**: コード変更 / レポート / 設計案 / レビュー結果
- **完了条件**: この分析から導出した明確な達成基準（後続フェーズで参照）
- **選択理由**: なぜこの種類・規模と判断したか 1〜2 文で記録

### ステップ 2: 戦略選択

以下の判定フローで戦略を決定する。**ユーザーへの確認は不要。**

```
タスクは複数の独立した作業に分解できるか？
├─ Yes → 戦略 B: Divide（並列実行→統合）
└─ No
    正解が 1 つでない、多様な視点が必要か？
    ├─ Yes → 戦略 C: Brainstorm（多角的アイデア収集）
    └─ No
        既存コード変更のレビューか？
        ├─ Yes → 戦略 D: Review（品質検証）
        └─ No → Commander-delegated Single（下記参照）
```

迷った場合の指針:
- 実装タスクで 2 つ以上の独立した変更点がある → **B (Divide)**
- 「どうすべきか」「どれが良いか」系の問い → **C (Brainstorm)**
- 「レビューして」「チェックして」系の指示 → **D (Review)**
- 上記に当てはまらない単一タスク → **Commander-delegated Single**

---

### Commander-delegated Single

Commander が分析した結果 Single モードが最適と判断した場合、ステップ 1 のコンテキストを引き継いで Single を実行する。

**手順**:
1. ステップ 1 の分析結果（種類・規模・成果物・完了条件・選択理由）をプロンプトに組み込む
2. エージェント選定基準表 + エージェントプロファイル表で最適なエージェントを選択（デフォルト claude ではなく、タスクの性質に応じて判断）
3. 完了条件をステップ 1 の分析結果から導出してプロンプトに明記
4. Single モードの Serial/Parallel 判定 → 実行フローに進む

---

### 戦略 B: Divide — タスク分解→並列実行→統合

**選択条件**: タスクが複数の独立した作業に分解できる。

**B-1. 分解** — 各サブタスクについて以下を決定:
- サブタスク名
- エージェント種別（「エージェント選定基準」+「エージェントプロファイル」を参照）
- プロンプト（「サブタスク プロンプトテンプレート」を使用）

**B-2. 並列起動**
```bash
cmux-delegate start <agent1> "<prompt1>"
# → TASK_ID, SURFACE_REF を記録

cmux-delegate start <agent2> "<prompt2>"
# → TASK_ID, SURFACE_REF を記録
```

起動状況をユーザーにテーブル形式で報告する。

**B-3. 結果収集** — 各タスクに対して `wait-poll` → `read` を順次実行（エージェントプロファイルのタイムアウト・interval を適用）。タイムアウト時のフォールバック（probe → 追加待機 or 途中結果取得）も適用する。

**B-4. 統合レポート** — 5 ステップの具体手順:

1. **結果収集チェック**: 各サブタスクのステータスを確認（COMPLETED / PARTIAL / FAILED）。FAILED のタスクは明示し、成功分のみで統合を進める。
2. **キーファインディング抽出**: 各サブタスクの結果から重要な発見を箇条書きで抽出。各項目に出典（サブタスク名）を付記。
3. **クロスリファレンス**: 複数のサブタスクの結果を突き合わせ、以下に分類:
   - **高信頼度**: 複数のサブタスクで一致した発見
   - **要確認**: 片方のみの発見、または曖昧な結果
   - **矛盾**: サブタスク間で相反する結果（競合ファイル変更、矛盾する設計判断など）
4. **ギャップ分析**: ステップ 1 の完了条件と突き合わせ、未達成の項目を洗い出す。
5. **統合レポート出力**:
```
## 統合結果

### 完了状況
| サブタスク | ステータス | エージェント |
|---|---|---|
| ... | COMPLETED/PARTIAL/FAILED | claude/codex/... |

### キーファインディング
- [高信頼度] ... （出典: サブタスク A, B）
- [要確認] ... （出典: サブタスク A）
- [矛盾] ... （A: ..., B: ...）

### ギャップ
- 未達成: ...

### 次のアクション
- ...
```

**B-5. クリーンアップ** — 全タスクに対して `cleanup`（エージェントプロファイルの猶予を適用）

---

### 戦略 C: Brainstorm — 同一テーマで多角的にアイデアを収集

**選択条件**: 正解が 1 つでない問題。設計方針、命名、アーキテクチャ選定など。

**C-1. プロンプト構築** — 共通のベースプロンプトに、各エージェントへ**異なる視点や制約**を付与する。視点はタスクの性質に応じて Commander が決定する。

例: アーキテクチャ検討
- エージェント A: 「シンプルさ・保守性を重視して」
- エージェント B: 「パフォーマンス・スケーラビリティを重視して」

例: API 設計
- エージェント A: 「利用者の使いやすさを重視して」
- エージェント B: 「内部実装のシンプルさを重視して」

**C-2. 並列起動** — 2〜3 エージェントを起動（異なるエージェント種別の組み合わせ推奨）

**C-3. 結果収集** — `wait-poll` + probe フォールバックを適用

**C-4. 統合レポート** — 5 ステップの具体手順:

1. **結果収集チェック**: 各エージェントのステータス確認。
2. **提案抽出**: 各エージェントの提案を構造化して抽出（視点・提案内容・根拠・トレードオフ）。
3. **クロスリファレンス**:
   - **共通ポイント**: 複数エージェントで一致した提案や認識
   - **独自ポイント**: 片方のみが挙げた視点
   - **対立ポイント**: 相反する提案とその根拠
4. **ギャップ分析**: ステップ 1 の完了条件と突き合わせ。
5. **統合レポート出力**:
```
## Brainstorm 統合結果

### 各エージェントの提案
| エージェント | 視点 | 提案概要 |
|---|---|---|
| A (claude) | シンプルさ重視 | ... |
| B (codex) | パフォーマンス重視 | ... |

### 共通ポイント
- ...

### 対立ポイント
| 論点 | エージェント A | エージェント B |
|---|---|---|
| ... | ... | ... |

### Commander の推薦
- 推薦案: ...
- 理由: ...
```

**C-5. クリーンアップ**

### 戦略 D: Review — コード変更の品質検証

**選択条件**: 実装済みのコード変更をレビューする。Divide の後に連鎖実行することもある。

**D-1. 差分取得**
```bash
git diff <base_ref>
git diff --name-only <base_ref>
```
- `base_ref` 未指定時: ステージされた変更があれば `--cached`、なければ `HEAD~1`
- 差分が空なら「レビュー対象の変更がありません」と報告して終了

**D-2. レビュープロンプト構築** — 以下の 5 観点を含む:
1. **バグ**: ロジックエラー、エッジケースの未処理、nil/null 参照
2. **セキュリティ**: インジェクション、認証漏れ、機密情報の露出
3. **パフォーマンス**: N+1 問題、不要なアロケーション、計算量の問題
4. **設計**: SOLID 原則、責務の分離、インターフェース設計
5. **コード規約**: 言語のイディオム、命名規則、エラーハンドリング

出力形式を指定: 各指摘を `ファイル:行 [観点] 重要度(BLOCKING/WARNING/INFO) - 説明` で出力し、最後に総合判定（LGTM / NEEDS_FIX / BLOCKING）を出力するよう指示する。

**D-3. エージェント起動**
- **codex は必ず含める**（コード変更のレビューでは codex の視点が必須）
- 差分 50 行以上: codex と claude の 2 エージェントで並列レビュー
- 差分 50 行未満: codex 1 エージェントのみ

**D-4. 結果収集** — `wait-poll` + probe フォールバックを適用

**D-5. 統合レビューレポート** — 5 ステップの具体手順:

1. **結果収集チェック**: 各レビューアのステータス確認。
2. **指摘抽出**: 各レビューアの指摘を `ファイル:行 [観点] 重要度 - 説明` 形式で正規化。
3. **クロスリファレンス**:
   - **高信頼度**: 両者共通の指摘 → 優先表示
   - **参考**: 片方のみの指摘 → 参考として表示
   - **矛盾**: 相反する指摘 → 両方の根拠を提示
4. **ギャップ分析**: 5 観点のうちカバーされていない観点がないか確認。
5. **統合レポート出力**:
```
## レビュー統合結果

### 指摘一覧
| # | ファイル:行 | 観点 | 重要度 | 信頼度 | 説明 |
|---|---|---|---|---|---|
| 1 | ... | バグ | BLOCKING | 高信頼度 | ... |
| 2 | ... | 設計 | WARNING | 参考 | ... |

### 総合判定: LGTM / NEEDS_FIX / BLOCKING
- 判定根拠: ...

### 推奨アクション
- ...
```

総合判定ロジック: 両者 LGTM → LGTM / いずれか BLOCKING → BLOCKING / それ以外 → NEEDS_FIX

**D-6. クリーンアップ**

---

### 戦略の連鎖

Commander は戦略を連鎖させることができる。前の戦略の結果を次の戦略のプロンプトに組み込む。

典型パターン:
- **Divide → Review**: 実装後に変更をレビュー
- **Brainstorm → Single**: アイデア出し後、採用案を 1 エージェントで実装
- **Brainstorm → Divide**: アイデア出し後、採用案を分割実装

連鎖の判断基準:
- Divide で実装した場合 → Review を**自動連鎖**する（明示的に不要と指示されない限り）
- Brainstorm の場合 → ユーザーに採用案を確認してから次の戦略へ進む

---

### エージェント選定基準

| エージェント | 特性 | 選定場面 |
|---|---|---|
| **codex** | コード変更に強い | 新機能追加、リファクタリング、バグ修正 |
| **claude** | 分析・設計に強い | 設計、レビュー、調査、ドキュメント作成 |
| **gemini** | 高速ワンショット | 短い質問、要約、簡単な変換 |
| **cmd** | 任意コマンド実行 | `go test ./...`, `npm run build`, リンター |

**必須ルール**:
- **実装を伴うタスクでは codex を第一候補とする。**
- **レビュー（戦略 D）では codex を必ず含める。**
- **プラン策定（Commander ステップ 1〜2）では codex を含めることを推奨する。**

### サブタスク プロンプトテンプレート

委任先は独立したセッションで起動される。十分なコンテキストをプロンプトに含めること:

```
## 親タスク
<Commander の分析概要: 種類・規模・全体目標>

## コンテキスト
- リポジトリ: <repo_path>
- ブランチ: <current_branch>
- 関連ファイル: <file_list>

## タスク
<具体的な指示>

## スコープ制約
- 上記タスクの範囲外の変更は行わないこと
- 関連する発見（範囲外だが重要な情報）は「追加発見」セクションに記載すること

## 完了条件
<明確な達成基準>

## 出力形式
以下の形式で結果を出力すること:

### 結果サマリ
<1〜3 文の要約>

### 詳細
```

**エージェント別補足**:
- **codex**: `git diff` の出力を結果に含めるよう指示を追加
- **gemini**: プロンプトを簡潔版にする（親タスク・スコープ制約を省略可）
- **cmd**: 出力形式の指示は不要（コマンドの stdout がそのまま結果）

### エラーハンドリング

| 状況 | 対応 |
|---|---|
| `start` 失敗 | 1 回リトライ → 失敗ならスキップし報告 |
| `wait-poll` タイムアウト | `probe` で状態確認 → active なら追加 60s → frozen/stalled なら途中結果取得 → idle なら完了扱い → unknown なら read 試行 |
| `probe` で frozen 検出 | `read` で途中結果を取得し「部分完了」として扱う → `cleanup` |
| `probe` で idle 検出 | 完了とみなし `read` → `cleanup`（codex 固有） |
| 一部タスク失敗 | 成功分のみで統合レポートを作成し、失敗タスクを明示 |
| 全タスク失敗 | エラー原因を分析して報告し、手動での対応を提案 |
| 同一ファイルの競合変更 | 統合レポートで競合箇所を明示し、手動マージを提案 |
| 結果末尾が不完全 | claude/codex: 追加 5s 待ち + 再 read |

## 注意事項

- `claude` / `codex` エージェントはスキップパーミッションで起動される — 信頼できるプロンプトのみ委任すること
- 並列数は最大 4 ペインを目安とし、タスクに応じて Commander が判断する
- 長時間タスクでは `cmux-delegate status <task_id> "進捗メッセージ"` でステータスを更新できる
- Review の統合判定は保守的に行う（疑わしい場合は NEEDS_FIX）
- 各フェーズの進捗をユーザーにリアルタイムで報告すること
- `wait-poll` の PROGRESS 出力は進捗テーブルとしてユーザーに提示すること
